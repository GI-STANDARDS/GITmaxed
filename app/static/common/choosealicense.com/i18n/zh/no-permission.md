当你创作一件创意作品（其中包括代码）时，该作品默认受排他性版权保护。除非你附上一份另作规定的许可证，否则任何其他人都不能复制、分发或修改你的作品，否则将面临下架、勒索或诉讼的风险。一旦作品有了其他贡献者（每个人都是版权持有者），「任何人」就开始把你也包括进去了。

即便没有许可证文件，在某些情况下你也可能授予了一些权利——当你将源代码发布到一个要求接受服务条款的网站时。例如，如果你将源代码发布到 GitHub 上的公开仓库，你就已经接受了[服务条款](https://help.github.com/articles/github-terms-of-service)，并由此允许他人查看和复刻你的仓库。如果[版权的限制与例外](https://en.wikipedia.org/wiki/Limitations_and_exceptions_to_copyright)适用于他人的特定情况，他们也可能无需你的许可。无论是网站条款，还是特定司法管辖区的版权限制，都不足以支撑人们通常在公共代码托管平台上所寻求的那类协作，比如开源许可证所促成的试验、修改和分享。

你无需做任何事就能*不*提供许可证。不过，你也许希望在一个显眼的位置（例如项目的 README）添加版权声明，并声明你不提供任何许可证，以免[用户](#for-users)误以为你是疏忽了。如果你打算接受他人对你这个无许可证项目的贡献，你或许可以和你的律师一起考虑为项目添加一份贡献者协议，这样即使你不授予相同的权利，也能保有来自贡献者的版权许可。

禁止他人使用你的代码，未必是你所说的「无许可证」的本意。[开源许可证](/)允许他人重用你的代码，同时保留版权。如果你的目标是彻底放弃版权限制，请改用[公共领域贡献声明](/licenses/#unlicense)。

## 致用户
{: #for-users }

如果你发现某个软件没有许可证，这通常意味着你未获得该软件创作者的任何许可去使用、修改或分享该软件。尽管像 GitHub 这样的代码托管平台可能允许你查看和复刻代码，但这并不意味着你被允许出于任何目的使用、修改或分享该软件。

你的选择：

{: .bullets}

- **礼貌地请求维护者添加许可证。**除非软件有强烈的相反迹象，否则缺少许可证很可能只是疏忽。如果软件托管在 GitHub 这样的网站上，请开一个 issue 请求添加许可证，并附上指向本站的链接。如果你比较大胆，而且哪种许可证最合适也相当明显，那就开一个 pull request 来添加许可证——参见本站每个许可证页面侧边栏中的「推荐此许可证」（例如 [MIT](/licenses/mit/#suggest-this-license)）。
- **不要使用该软件。**寻找或创建一个采用开源许可证的替代品。
- **协商一份私有许可证。**带上你的律师。
